# 竞赛级传输策略实现文档

## 1. 概述

本文档描述了一套面向网络模拟竞赛的高性能传输策略实现。该策略采用**事件驱动架构**，集成 **SACK 精确恢复**、**交换机 SRPT 严格优先级调度** 与 **ECMP 动态负载均衡**，在完全遵循 `sim_api` 接口约束的前提下，实现了低延迟、高吞吐、 deadline 感知的网络传输。

## 3. 核心常量与参数

### 3.1 基础常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `MSS` | `MAX_PAYLOAD` | 最大段大小（字节），由模拟器定义 |
| `HEADER` | 64 | 包头大小（字节） |
| `MIN_CWND` | 2 x MSS | 最小拥塞窗口 |
| `INIT_CWND_RATIO` | 2.0 | 初始窗口相对于 BDP 的倍数 |
| `MIN_RTO` | 200.0 us | 最小重传超时 |
| `MAX_RTO` | 1,000,000.0 us | 最大重传超时 |
| `DEFAULT_RTO` | 1000.0 us | 默认重传超时（无样本时） |
| `DELAYED_ACK_US` | 5.0 us | 延迟 ACK 定时器间隔 |

### 3.2 可调参数说明

| 参数 | 调优方向 | 影响 |
|------|----------|------|
| `INIT_CWND_RATIO` | 提高初始突发 | 大 BDP 网络降低慢启动耗时 |
| `DELAYED_ACK_US` | 更频繁 ACK | 更快反馈，但增加控制开销 |
| `MIN_CWND` | 避免过度保守 | 小流快速完成 |
| `scan_limit` (交换机) | 扫描更多包 | 更精确 SRPT，但增加 CPU 开销 |

## 4. 发送端实现（`SenderFlow`）

### 4.1 状态结构

```python
class SenderFlow:
    fid: int              # 流 ID
    dst: int              # 目的节点
    total: int            # 总字节数
    prio: int             # 流优先级
    deadline: float       # 截止时间 (us)

    next_new: int         # 下一新数据字节序号
    acked: int            # 累积确认序号
    cwnd: int             # 拥塞窗口（字节）
    ssthresh: int         # 慢启动阈值

    inflight: List[dict]  # 未确认段 [{seq, len, time, retx}]
    retx_queue: List[dict] # 待重传段

    srtt: float           # 平滑 RTT
    rttvar: float         # RTT 方差
    rto: float            # 当前 RTO 值
    rto_active: bool      # RTO 定时器是否激活
    rto_expire_time: float # RTO 到期时间（用于防早触发）

    last_ack: int         # 上一个 ACK 序号
    dup_cnt: int          # 重复 ACK 计数
    recovery: bool        # 是否处于快速恢复
    recover_point: int    # 恢复点序号
```

### 4.2 BDP 感知初始窗口

在流到达时根据链路带宽和 RTT 计算 BDP：

```
BDP = bw_bytes_us x base_rtt_us
初始窗口 cwnd = min(total_size, max(MIN_CWND, BDP x INIT_CWND_RATIO))
```

**设计意图**：避免小流过度保守，大流充分利用带宽时延积。

### 4.3 拥塞控制状态机

```
                    Slow Start
                    cwnd < ssthresh
                    cwnd += acked
                           |
                           | cwnd >= ssthresh
                           v
                    Congestion Avoid
                     cwnd >= ssthresh
                    cwnd += MSS*acked/cwnd
                           |
                           | 3 Dup ACK
                           v
                    Fast Recovery
                    ssthresh=cwnd/2
                    cwnd=ssth+3*MSS
                           |
           New ACK > recover_point
                           |
                           v
                    Timeout
                    ssthresh=cwnd/2
                    cwnd = MIN_CWND
```

### 4.4 段调度逻辑

```python
def next_segment(self) -> dict:
    # 1. 优先重传队列（队首弹出）
    if self.retx_queue:
        seg = self.retx_queue.pop(0)
        seg['retx'] = True
        return seg

    # 2. 新数据（受 cwnd 限制）
    if self.can_send():
        length = min(MSS, total - next_new)
        seq = next_new
        next_new += length
        return {'seq': seq, 'len': length, 'retx': False}

    return None
```

### 4.5 ACK 处理流程

```
on_ack(ack_seq, sack_blocks, echo, now):

    1. 累积确认移除
       - 移除 seg_end <= ack_seq 的段
       - 测量 RTT（仅非重传段）

    2. SACK 精确移除
       - 遍历 SACK 块 [start, end]
       - 完全包含于 SACK 区间的段视为已确认
       - 测量 RTT（仅非重传段）

    3. 重复 ACK 检测
       - ack_seq == last_ack 且 newly_acked == 0: dup_cnt++
       - ack_seq > last_ack: dup_cnt = 0, 可能退出恢复

    4. 拥塞窗口更新
       - 恢复状态: cwnd += newly_acked（窗口膨胀）
       - 慢启动: cwnd += newly_acked
       - 拥塞避免: cwnd += max(1, MSS*newly_acked//cwnd)

    5. 快速重传触发
       - dup_cnt == 3 且未恢复: _fast_retransmit()
```

### 4.6 RTT 估计（RFC 6298）

```python
def _update_rtt(self, rtt: float):
    if srtt == 0:
        srtt = rtt
        rttvar = rtt / 2
    else:
        rttvar = 0.75 * rttvar + 0.25 * abs(srtt - rtt)
        srtt = 0.875 * srtt + 0.125 * rtt
    rto = clamp(MIN_RTO, srtt + 4*rttvar, MAX_RTO)
```

### 4.7 快速重传与恢复

```python
def _fast_retransmit(self):
    recovery = True
    recover_point = next_new          # 恢复点设为最高已发送序号
    ssthresh = max(MIN_CWND, cwnd // 2)
    cwnd = ssthresh + 3 * MSS         # 窗口膨胀

    # 重传最早未确认段
    unacked = sorted(inflight, key=lambda s: s['seq'])
    if unacked:
        lost = unacked[0]
        if not already_in_retx_queue:
            retx_queue.append(lost 去除 time 字段)
```

### 4.8 超时处理

```python
def on_timeout(self):
    # 所有未确认段加入重传队列
    for s in inflight:
        if not already_in_retx_queue:
            retx_queue.append(s 去除 time 字段)
    inflight.clear()

    ssthresh = max(MIN_CWND, cwnd // 2)
    cwnd = MIN_CWND
    recovery = False
    dup_cnt = 0
```

## 5. 接收端实现（`RecvContext` + Host 逻辑）

### 5.1 状态结构

```python
class RecvContext:
    fid: int      # 流 ID
    src: int      # 源节点（用于 ACK 寻址）
    timer_active: bool  # 延迟 ACK 定时器是否激活
```

### 5.2 数据包处理流程

```
on_packet_arrival(DATA pkt):
    1. 初始化/获取 RecvContext

    2. 顺序交付检查
       if pkt.seq_no == get_delivered_up_to(fid):
           deliver_packet(pkt.id)
       else:
           ensure_rx_space(pkt.size)   # 必要时丢弃低优先级包
           add_packet_to_buffer(pkt.id)

    3. 连续前缀推进
       loop:
           cur = get_delivered_up_to(fid)
           hit = 缓冲区中 seq_no == cur 的包
           if not hit: break
           deliver_packet(hit.id)

    4. 生成/缓存 ACK
       build_and_cache_ack(fid, echo_time)

    5. 启动延迟 ACK 定时器（若未激活）
       schedule_timer(DELAYED_ACK_US, ('dack', fid))
```

### 5.3 接收缓冲区管理

```python
def _ensure_rx_space_for(self, needed: int, ctx: HostContext):
    while get_rx_buffer_remaining() < needed:
        buf = get_received_packets()
        if not buf: break

        # 丢弃优先级最低的数据包（保留 ACK 和高优先级流）
        candidate = None
        worst_prio = -inf
        for pv in buf:
            if pv.is_ctrl: continue
            prio = pv.custom.get('p', 0)
            if candidate is None or prio < worst_prio:
                worst_prio = prio
                candidate = pv

        if candidate:
            drop_packet_in_buffer(candidate.packet_id)
        else:
            break
```

### 5.4 SACK 块生成

```python
def _build_and_cache_ack(self, fid, ctx, echo_time):
    delv = get_delivered_up_to(fid)

    # 1. 收集缓冲区中该流的乱序区间
    intervals = []
    for pv in get_received_packets():
        if pv.flow_id == fid and not pv.is_ctrl:
            start = pv.seq_no
            end = start + pv.payload_size
            if end > delv:
                intervals.append((start, end))

    # 2. 合并重叠/连续区间
    merged = []
    for s, e in sorted(intervals):
        if merged and s <= merged[-1][1]:
            merged[-1] = (merged[-1][0], max(merged[-1][1], e))
        else:
            merged.append((s, e))

    # 3. 截取从 delv 开始的块，最多 3 个
    sacks = [[max(s, delv), e] for s, e in merged if max(s, delv) < e][:3]

    # 4. 缓存到 ack_pool（按源节点索引，自动合并）
    ack_pool[src] = {
        'type': 'ack',
        'flow_id': fid,
        'ack_seq': delv,
        'sack': sacks,
        't': echo_time
    }
```

### 5.5 延迟 ACK 机制

- **目的**：减少控制包数量，降低反向链路负载
- **实现**：收到数据后启动 5us 定时器，到期后通过 `request_link_ready()` 触发 ACK 发送
- **合并**：同一流的多个数据包到达，ACK 缓存自动更新为最新状态

## 6. Host 策略调度（`MyHostStrategy`）

### 6.1 状态结构

```python
class MyHostStrategy:
    senders: Dict[int, SenderFlow]      # 发送流表
    recvs: Dict[int, RecvContext]        # 接收流表
    ack_pool: Dict[int, dict]           # 待发送 ACK 缓存 {dst_node: ack_data}
    bw_bytes_us: float                   # 链路带宽 (bytes/us)
    base_rtt_us: float                   # 基础 RTT (us)
```

### 6.2 发送调度优先级

```
调度顺序：
1. ACK 控制包（绝对优先）
   -> 从 ack_pool 弹出并发送

2. 数据包（SRPT + 优先级）
   -> 收集所有有可发数据的流
   -> 排序键: (-priority, remaining_bytes, deadline)
   -> 依次尝试发送
```

### 6.3 定时器管理

#### RTO 定时器

```python
if event == 'rto':
    sf = senders.get(fid)
    if sf and sf.rto_active:
        if now >= sf.rto_expire_time:
            # 真正超时
            sf.rto_active = False
            if sf.inflight:
                sf.on_timeout()
                request_link_ready()
                # 重启定时器
                sf.rto_active = True
                sf.rto_expire_time = now + sf.rto
                schedule_timer(sf.rto, ('rto', fid))
        else:
            # 防早触发：重新注册剩余时间
            remain = max(1.0, sf.rto_expire_time - now)
            schedule_timer(remain, ('rto', fid))
```

**设计要点**：使用 `rto_expire_time` 字段防止定时器早触发（因 `schedule_timer` 可能不支持取消）。

#### 延迟 ACK 定时器

```python
if event == 'dack':
    rc = recvs.get(fid)
    if rc: rc.timer_active = False
    request_link_ready()  # 触发 ACK 发送
```

## 7. 交换机实现（`MySwitchStrategy`）

### 7.1 状态结构

```python
class MySwitchStrategy:
    next_hops: Dict[int, List[LinkId]]   # 目的节点 -> 候选端口列表
```

### 7.2 ECMP 动态选路

```python
def on_packet_arrival(pkt, input_port, ctx):
    candidates = next_hops.get(pkt.dst_node, [])

    best_port = None
    min_usage = inf

    for port in candidates:
        usage = ctx.get_port_usage(port)
        if usage.capacity_bytes - usage.used_bytes >= pkt.size:
            if usage.used_bytes < min_usage:
                min_usage = usage.used_bytes
                best_port = port

    return best_port  # None 表示丢包
```

**策略**：选择当前队列占用字节最少的端口（Least Queue Usage, LQU）。

**特点**：
- 无需 RED 参数，简化配置
- 动态适应流量变化
- 队列满时自然丢包（返回 None）

### 7.3 SRPT + 严格优先级调度

```python
def on_link_ready(output_port, ctx):
    ids = ctx.get_port_queue_packet_ids(output_port)
    if not ids: return None

    scan_limit = min(len(ids), 100)  # 限制扫描范围

    best_id = ids[0]
    best_score = _packet_priority(ctx.get_packet_view(best_id))

    for i in range(1, scan_limit):
        score = _packet_priority(ctx.get_packet_view(ids[i]))
        if score < best_score:
            best_score = score
            best_id = ids[i]

        # 优化：找到控制包立即返回
        if best_score[0] == 0:
            break

    return best_id
```

### 7.4 包优先级评分函数

```python
def _packet_priority(pv: PacketView) -> tuple:
    # 返回值越小，优先级越高
    if pv.is_ctrl:
        return (0, 0, 0, 0)  # ACK 绝对优先

    c = pv.custom
    if isinstance(c, dict):
        prio = c.get('p', 0)      # 流优先级
        rem = c.get('r', 10**9)   # 剩余字节（SRPT）
        dl = c.get('d', 10**9)    # 截止时间
        return (1, -prio, rem, dl)

    return (1, 0, 10**9, 10**9)  # 默认低优先级
```

**排序语义**：

| 维度 | 说明 |
|------|------|
| (0, ...) vs (1, ...) | 控制包优先于数据包 |
| -prio | 高优先级流优先（数值越大越优先） |
| rem | 剩余字节少的优先（SRPT，最短剩余处理时间） |
| dl | 截止时间近的优先 |

## 8. 关键设计决策

### 8.1 为什么选择 LQU 而非加权 ECMP？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **LQU（当前）** | 简单、无需带宽参数、动态适应 | 未考虑端口带宽差异 |
| 加权 ECMP | 考虑带宽差异 | 需要维护带宽表，配置复杂 |

**决策**：LQU 在模拟器环境中足够有效，且避免了旧版 RED 的参数调优问题。

### 8.2 为什么移除 RED？

旧版使用 RED 主动队列管理，但：
- 需要精细调参（W, min_th, max_th, max_p）
- 模拟器环境队列行为可能与真实网络不同
- LQU 在队列满时自然丢包，等效于尾部丢弃

**决策**：简化交换机逻辑，依赖 LQU 和优先级调度实现拥塞控制。

### 8.3 延迟 ACK 的 5us 设置

- **权衡**：过短（如 0us）= 每个包一个 ACK，控制开销大
- **权衡**：过长（如 40ms）= 反馈延迟，降低吞吐
- **选择**：5us 在模拟器时间尺度下平衡了及时反馈与开销

### 8.4 扫描限制 100 的设计

```python
scan_limit = min(len(ids), 100)
```

- **目的**：避免大队列时的 O(n) 全量扫描
- **效果**：前 100 个包中大概率包含高优先级包（因优先级调度会快速排空高优先级包）
- **权衡**：极端情况下可能错过队列深处的高优先级包

## 10. 性能优化要点

### 10.1 发送端优化

1. **BDP 感知初始窗口**：避免慢启动耗时，快速达到稳定速率
2. **SACK 精确恢复**：避免不必要的超时重传，提高带宽利用率
3. **防早触发 RTO**：`rto_expire_time` 字段确保定时器精确性
4. **流级 SRPT 调度**：高优先级、短流优先，降低平均完成时间

### 10.2 接收端优化

1. **延迟 ACK**：5us 合并减少控制包数量
2. **优先级感知丢弃**：缓冲区不足时丢弃低优先级流，保护高优先级流
3. **连续前缀自动推进**：自动交付乱序到达的连续段

### 10.3 交换机优化

1. **LQU ECMP**：动态负载均衡，避免热点端口
2. **SRPT 调度**：最短剩余处理时间优先，降低流平均完成时间
3. **控制包绝对优先**：ACK 零延迟转发，确保拥塞反馈及时性
4. **扫描优化**：限制扫描范围，平衡调度精度与计算开销

## 11. 已知限制与改进方向

| 限制 | 影响 | 改进方案 |
|------|------|----------|
| 无显式流量控制 | 接收端可能溢出 | 添加接收窗口通告（rwnd） |
| 固定 MSS | 未适应路径 MTU | 动态探测 MSS |
| 交换机无带宽感知 | 异构链路负载不均 | 加权 LQU（usage/bw） |
| 扫描限制 100 | 极端情况调度次优 | 自适应扫描深度 |
| 单一 ACK 缓存 | 多流 ACK 可能延迟 | 按流独立定时器 |
| 无拥塞窗口上限 | 大 BDP 可能过度膨胀 | 添加 max_cwnd 限制 |
| 定时器精度依赖 | `schedule_timer` 实现 | 使用更精确的定时器机制 |

| 初始窗口 | 固定 10xMSS | BDP 自适应 |
| 交换机调度 | 严格优先级（仅 prio） | SRPT + 优先级 + deadline |
| ECMP | 加权占用（考虑带宽） | LQU（最小队列占用） |
| RED | 显式 RED 算法 | 移除（依赖 LQU 自然丢包） |
| 延迟 ACK | 未实现 | 5us 延迟 ACK |
| RTO 管理 | 简单定时器 | 防早触发精确管理 |
| 接收缓冲丢弃 | FIFO | 优先级感知丢弃 |
| 扫描优化 | 无 | 限制 100 包扫描 |
