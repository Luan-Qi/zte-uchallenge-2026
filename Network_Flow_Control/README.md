# 高性能网络流量调度仿真

## 1.赛题背景：

在现代数据中心中，网络不再仅仅是连接服务器的管道，而是成为了计算性能的瓶颈。根据SIGCOMM和NSDI等顶级会议的研究，AI训练任务中的通信操作会产生巨大的同步流量突发，如果处理不当，会导致交换机缓冲区瞬间溢出，进而引发丢包。在传统的TCP协议中，丢包会触发超时重传（RTO），导致毫秒级的时延，这对于需要微秒级同步的分布式系统是不可接受的。新型协议如RoCE（RDMA over Converged Ethernet），通过PFC（Priority Flow Control）机制实现了无损网络。然而，PFC容易引发队头阻塞和拥塞扩散，甚至导致死锁。

因此，如何在网络上通过智能的路径选择、端到端拥塞控制和交换机的主动队列管理来实现低时延、高吞吐，成为了当前的研究热点。

本赛题正是基于这一背景设计的。我们将复杂的物理网络抽象为一个离散事件仿真器，保留了核心的物理属性（如排队时延、传输时延、缓冲区容量），隐去了物理层的电气细节。参赛者的代码将充当网络节点的“大脑”，直接决定每一个数据包的命运。

## 2.**赛题简介**：进行赛题抽象描述

随着大模型训练、高性能计算以及分布式存储系统的爆发式增长，数据中心网络正面临前所未有的性能挑战。传统的网络传输协议（如TCP/IP）和静态的流量调度策略，在面对微秒级低时延需求和百G级高吞吐量需求时，往往显得力不从心。特别是在处理“大象流”与“老鼠流”共存、多路径负载均衡以及突发性拥塞控制等问题上，工业界和学术界正迫切寻求突破性的解决方案。智能网卡以及可编程交换机（P4）等技术的兴起，标志着网络控制逻辑正从内核向应用层、从固定硬件向可编程硬件迁移。

本赛题要求参赛者在一个高保真的离散事件仿真环境中，接管网络传输的全流程控制权。参赛者不仅需要设计端到端的发送与接收策略，还需要编写中间交换节点的调度逻辑，保证数据高效传输。通过模拟真实网络环境中的带宽限制、传播时延、缓冲区溢出以及多路径路由选择。本赛题核心要求如下：

1. **全栈控制**： 参赛者需同时控制发送端（报文调度、报文标识、资源抢占）、网络端（队列管理、多路径路由）和接收端（接收确认、乱序重排、丢包重传）。
1. **现实约束**： 引入物理层的硬性约束，包括链路带宽上限、交换机缓存限制、传播延迟等。
1. **多目标优化**： 在保证高吞吐的同时，最小化业务完成时间，并处理乱序和丢包带来的性能抖动。

## 3.赛题模型

### 3.1 业务传输：

业务流由APP 产生，通过网卡发送，在网络中以报文形式传输，网卡会将业务流切割成多个有序的报文，按序发送。报文大小可由选手策略自定义，但不超过最大传输单元（MTU）。报文通过网络传输到接收端网卡时，如果未产生乱序，则可直接提交到接收端APP；如果乱序到达，需要暂存在接收端网卡，进行保序处理后才能提交到APP。

节点发包逻辑：每个节点有多条链路，与其他节点连接，每条链路独立发包。链路每个时间点只能发送一个报文，报文占用链路的时间即为传输时延。链路发送完报文后，会立刻进行下一个报文的调度。

### 3.2 时延构成模型

仿真中，我们简单将一个数据包从源节点到目的节点的端到端时延 T_total 分为以下几部分：

    T_total = T_prop + T_trans + T_queue

1. 传播时延 (T_prop)：信号在物理介质中传播的时间，取决于链路长度和光速。赛题中给定的静态拓扑参数（Latency）。

2. 传输时延 (T_trans)：将数据包推上链路所需的时间，取决于链路带宽。

    T_trans =报文大小 (bits)/链路带宽(bps)

3. 排队时延 (T_queue)：数据包在交换机/网卡输出端口缓冲区等待发送的时间。这是唯一可以通过算法优化的变量，也是拥塞控制的核心战场。

    T_queue=Current Queue Length/Link Bandwidth

4. 其他时延：本赛题中忽略不计

 

### 3.3 缓冲区丢包模型

网络侧：网络交换机为每条链路的输出端口维护一个有限大小的队列。当队列缓存达到缓冲区上限时，新到达的数据包将被丢弃。

参赛者需要实现拥塞丢弃策略，这不仅仅是被动的尾部丢弃，更包括主动队列管理。

举例：RED（Random Early Detection）算法定义了丢包概率 p 与平均队列长度 avg_q 的关系 ：

- avg_q <= min_th ： p =  0
- min_th < avg_q < max_th：p = (avg_q- min_th)/(max_th - min_th)*max_p 
- avg_q > max_th： p = 1

参赛者需要在交换机节点策略中实现类似逻辑，或者设计基于优先级的丢弃策略（例如，当缓冲区满时，丢弃低优先级报文以容纳高优先级报文）。

网卡侧：网卡发送端无缓存大小限制，接收端给定一个总缓存限制，接收端缓存主要用于乱序重排，接收端乱序重排需要为每个乱序流维护一个重排缓存区，当所有流的缓存区之和大于网卡总缓存时，新到达的数据包将被丢弃。

参数者需要实现乱序重排策略。一方面需要减小乱序规模，另一方面需要处理丢包和乱序的逻辑。

### 3.4 丢包重传机制

报文在网络传输中，会因为拥塞或链路异常导致丢包，接收端发现链路丢包时，需及时通知发送端进行报文重传。

### 3.5 拥塞流控机制

网络中流量传输会因流量汇聚、微突发、负载不均导致拥塞，拥塞报文积压在网络交换机缓存中，当交换机缓存不足将会导致丢包。

参赛者可以自定义拥塞流控算法，如在传输的报文中携带拥塞信息，让网卡利用拥塞信息进行发送速率控制，避免拥塞，也可以允许报文丢弃，利用丢包重传机制重发报文。

### 3.6 节点通信机制：

整个组网属于分布式体系，即每个节点仅能够通过报文进行通信，不存在共享数据或中央控制器同步等机制。

参赛者可以通过报文的自定义字段或者自定义协议包，进行节点间的信息同步。

## 四、选手任务

选手需要实现并注入三类策略：Host策略（包括发送端和接收端）、交换机策略。

### 4.1 发送端策略

#### 需要做什么

- 当有新流到达本主机时，为该流初始化内部状态（例如下一段要发的字节位置、拥塞窗口、慢启动阈值、在途字节数、重传队列等），并可根据流的大小、优先级、截止时间等选择不同控制方式。
- 当本主机收到来自网络的确认包（ACK/SACK）时，根据确认的字节范围、选择性确认块和是否被标记拥塞，更新该流的已确认进度、在途字节数和重传队列，并据此调整拥塞窗口；在合适的时候请求本机网卡尽快进行下一次发送决策。
- 当本主机网卡空闲、仿真器询问“下一步发什么”时，在所有未完成流中选出一条流，再在该流上决定是发重传数据还是新数据、以及本段长度和优先级，并返回这一段数据的描述（流标识、目的节点、起始字节、本段长度、优先级）；若当前不应发任何包，则返回“不发”。
- 所有基于时间的逻辑（例如超时重传、按时间节奏发包）都通过“注册定时器、在定时器到期时执行相应动作”的方式实现，由选手在策略内自行设计。

#### 可以做什么

- 在多流之间按优先级、公平性、截止时间等做调度（例如优先发高优先级流或临近截止时间的流）。
- 实现不同的拥塞控制思路（如加性增乘性减、对拥塞标记的响应、与延迟 ACK 配合的节奏控制等）。
- 在首传按序、重传可重复已发未确认区间的约束下，设计重传策略（例如仅超时重传、或结合选择性确认做快速重传）。

例子：为每条流维护拥塞窗口和慢启动阈值，新流到达时初始化这些状态；每次网卡空闲时，在所有未完成流里按优先级选一条，若该流有需要重传的段则先发重传，否则在窗口允许范围内发下一段新数据；收到确认后根据确认的字节推进已确认位置、缩小在途字节数并更新窗口，若发现某段长时间未确认则加入重传并在下次发送时优先发送。超时重传通过“在发出未确认数据时注册一个定时器，到期后检查仍未确认的段并加入重传队列、缩小窗口”来实现。


### 4.2 接收端策略

#### 需要做什么

当有数据包成功写入本主机的接收缓冲后（即不是重复包、也未因缓冲满被丢弃），根据当前该流的接收缓冲状态，决定是否立即向发送端反馈确认。
通过仿真器提供的接口读取“该流当前已到达但尚未交付的字节区间”，据此计算出“已连续交付到的字节位置”以及其后的空洞区间，得到确认序号和可选的选择性确认块。
在决定发送确认时，使用仿真器提供的接口构造确认包（包含流标识、确认序号、选择性确认块、是否被标记拥塞等），并将该确认包提交给仿真器发送回源主机。

#### 可以做什么

- 采用“每收到一个数据包就发一个确认”，或“延迟确认”：收到包后先不立刻发确认，而是启动一个短定时器，若在定时器到期前没有新包到达，再发一个确认以合并多次接收。
- 在计算确认序号和选择性确认块时，可同时参考“已交付到的字节位置”与“已到达但未交付的区间”，以更准确反映接收进度。

例子：每次有数据包写入接收缓冲后，读取该流当前“已到达但尚未交付”的区间列表，按起始位置排序并合并重叠或相邻区间；从 0 开始沿这些区间推进，直到遇到空洞为止，得到最大连续前缀的结束位置作为确认序号；将确认序号之后的不连续区间作为选择性确认块（数量可做上限）；然后构造并发送一个包含该确认序号和这些块的确认包。若希望做延迟确认，可以在收到包时不立即发送，而是注册一个短延时定时器，在定时器到期时再按同样方法计算并发送确认。

### 4.3 交换机策略

#### 需要做什么

- 当有数据包到达本交换机时，根据包的目的节点和本交换机的转发表信息，在若干候选输出端口中选择一个端口，将包送入该端口的队列；若队列未满则包会入队等待发送，若满则包被丢弃。也可以选择不将包送入任何端口（即主动丢弃，例如做拥塞控制或过滤）。
- 当某个输出端口对应的链路空闲、仿真器询问“从该端口队列中下一个发哪个包”时，根据该端口当前的队列视图（各优先级下的排队包信息），选出一个包发送；若本次选择不发送，则端口保持空闲，直到后续再次被调度。

#### 可以做什么

- 在多个候选下一跳中选择端口时，结合各端口的队列占用或历史负载做负载均衡或拥塞感知选路。
- 在从队列中选包时，实现严格优先级（高优先级非空则只发高优先级）、加权轮询、基于流的公平队列等。
- 在合适时机主动从队列中丢弃某些包（例如队列超过门限时丢新到的包或队尾的包），以实现类似 AQM 的行为。

例子：包到达时，计算包从哪个端口输出能最快到达目的，选择将包送入该端口队列；当某端口链路空闲时，从端口队列中选择优先级最高的包发送。 

## 5. 评分规则

具体评分规则以附件中的 README为准，此处进行简要介绍。

### 5.1 输入与约定

一个数据集包含多条流，每条流包含：

- 权重：该流在总分中的相对重要程度，为正数。
- 开始时间：该流生成时刻。
- 截止时间：该流必须在此时间前完成，否则本条流得分为 0。

### 5.2 单条流得分

对于在截止时间前完成的流，其得分为：

$$
流得分 = 权重 \times \min\left(1, \frac{\text{实际完成时长}}{\text{期望完成时长}}\right)
$$

期望完成时长的计算：
- 记L=源到目的的最短传播时延（单位微秒）
- 记C=源到目的的最大流容量（字节/微秒），即最小割的总带宽。
- 期望完成时长= L + 流字节数/C。

### 5.3 单数据集得分

基准分=min(100.0, 每条流得分之和/流权重之和×100.0)

完成率=成功完成的流/流总数

但数据集得分=基准分×完成率^2

说明：选手应尽量保证完成率，避免最终得分被拉低。

### 5.4最终得分

线上测试有多个数据集，取各数据集得分的均值。

注意：

    (1) 保证所有流完整到达接收端，每条流报文保序。

    (2) 方案设计文档需要匹配代码实现。

    (3) 利用仿真器缺陷得到的分数无效

最终排名：跑分占比50%，代码实现占比30%，方案设计占比 20%
